# Обзор системы — Боттлнеки и нагрузочные тесты

## Краткое описание

| Параметр               | Значение                                          |
|------------------------|---------------------------------------------------|
| Скриптов               | 8 (4 биржи × 2 рынка)                             |
| Транспорт              | WebSocket (async, websockets>=12)                 |
| Хранилище              | Redis (async, redis>=7, pipeline батчи)           |
| Flush тикеров          | каждые 50ms                                       |
| Flush истории          | каждые 1s                                         |
| Хранение истории       | 5 чанков × 20 минут ≈ 1.5–2 часа                 |
| Хранение логов         | 2 чанка × 24 часа ≈ 48 часов                     |
| Ожидаемый throughput   | 500–3000 msg/s суммарно по всем скриптам          |
| Ожидаемая proc latency | 0.3–5ms (recv→redis)                              |

## Боттлнеки

### 1. Redis pipeline — основной боттлнек на запись

**Что:** Каждые 50ms скрипт делает `pipeline.execute()` с N командами `HSET`.

**Проблема:** При большом числе символов (500+) и высоком tick rate пайплайн может
накапливать отставание. Если выполнение `pipeline.execute()` занимает >50ms —
следующий flush уже стоит в очереди, буфер растёт.

**Признаки:**
- `proc latency p95 > 10ms` в снапшоте
- `t.writes/s` заметно ниже `messages/s`

**Решение:**
- Проверить `redis-cli --latency` на хосте Redis
- Перенести Redis на localhost, если сейчас сетевой
- Увеличить `TICKER_FLUSH_INTERVAL` до 100ms, чтобы собирать больший батч

---

### 2. WebSocket переподписка при реконнекте

**Что:** Bybit и Gate при реконнекте заново отправляют сотни subscribe-сообщений
батчами.

**Проблема:** Пока идёт подписка (несколько секунд), данные не поступают. При нестабильной
сети это означает регулярные дыры в истории.

**Признаки:**
- В снапшоте частые `reconnects > 0`
- Пропуски в `md:hist:bybit:*` списках

**Решение:**
- Мониторить `reconnects_total` в снапшоте
- При систематических дисконнектах проверить стабильность сети до биржи
- Для Gate уменьшить `SUB_BATCH` если биржа отвечает таймаутом на большие батчи

---

### 3. Конвертация символов OKX/Gate — пропуски при нестандартных quote

**Что:** `_to_okx()` и `_to_gate()` возвращают `None` для символов не на USDT/USDC.

**Проблема:** Если в subscribe-файле есть `BTCBTC`, `ETHBNB` и т.п. — они
**молча пропускаются** без ошибки. Данные по ним не собираются.

**Признаки:**
- `symbols_tracked` меньше числа строк в subscribe-файле
- Нет ключа `md:okx:spot:XYZBTC` в Redis

**Решение:**
- Держать в subscribe-файлах только USDT/USDC пары
- При необходимости — расширить функцию конвертации в скрипте

---

### 4. Memory growth при высоком tick rate

**Что:** `_history_buf[symbol]` накапливает записи между flush-циклами (каждые 1s).

**Проблема:** При 3000 msg/s на 500 символов за 1s накапливается ~6 записей на символ.
При зависании history flusher (например, Redis недоступен) буфер растёт бесконтрольно.

**Признаки:**
- RSS процесса растёт > 200MB
- `h.entries/s` = 0 при ненулевом `messages/s`

**Решение:**
- Контролировать RSS через `ps aux | grep market_data`
- Добавить cap на размер `_history_buf[symbol]` если нужно

---

### 5. Binance — отсутствие RTT метрики

**Что:** Binance `bookTicker` не включает exchange timestamp.

**Проблема:** Нельзя измерить реальную задержку от биржи до скрипта. Видна только
`proc latency` (буфер → Redis), которая не отражает сетевую задержку.

**Решение:** Для мониторинга сетевой задержки Binance — использовать внешний пинг до
`stream.binance.com`.

---

### 6. Gate.io — задержка при подписке (SUB_BATCH × 50ms)

**Что:** Gate требует 50ms паузы между батчами subscribe.

**Проблема:** При 500 символах и `SUB_BATCH=100` — 5 батчей × 50ms = **250ms**
до получения первых данных по последним символам после реконнекта.

**Признаки:** Первые данные по Gate появляются позже остальных после старта.

**Решение:** Это нормальное поведение, документировано. При необходимости увеличить
`SUB_BATCH` (если биржа это допускает).

## Нагрузочные тесты

Тесты находятся в `market_data/tests/load_test.py`. Запуск:

```bash
python3 market_data/tests/load_test.py
```

### Что тестируется

1. **Redis pipeline throughput** — скорость записи N тикеров через pipeline
2. **Redis read throughput** — скорость чтения HGETALL для N символов
3. **History RPUSH throughput** — скорость записи в List (история)
4. **SCAN throughput** — скорость сканирования `md:*` ключей
5. **Concurrent write** — параллельная запись из нескольких coroutine

### Интерпретация результатов

| Тест                    | Норма                | Проблема          |
|-------------------------|---------------------:|-------------------|
| Pipeline write (500 sym)| < 5ms                | > 20ms            |
| Read HGETALL (500 sym)  | < 10ms               | > 50ms            |
| History RPUSH (1000 ent)| < 10ms               | > 30ms            |
| SCAN md:* (10k keys)    | < 50ms               | > 200ms           |
| Concurrent write (8×500)| < 15ms/batch         | > 50ms/batch      |

Если тест показывает числа в зоне "Проблема":
- Проверить `redis-cli --latency`
- Проверить сетевой путь до Redis (`ping REDIS_HOST`)
- Проверить загрузку CPU/IO Redis-сервера (`redis-cli INFO stats`)

## Мониторинг в продакшне

### Ключевые метрики для алертинга

```
1. reconnects_total растёт > 5/мин     → нестабильный WS
2. proc latency p95 > 10ms             → Redis перегружен
3. msgs/s = 0 при статусе ACTIVE       → заморожен (редко, но бывает)
4. symbols_active/symbols_tracked < 90%→ часть символов не получает тики
5. chunk elapsed не растёт             → ротация сломана
```

### Проверка через redis-cli

```bash
# Сколько активных тикеров (должно = сумма символов по всем скриптам)
redis-cli --scan --pattern 'md:*:*:*' | grep -v hist | grep -v chunks | wc -l

# Последнее обновление конкретного тикера
redis-cli HGET md:binance:spot:BTCUSDT ts

# Размер истории для символа
redis-cli LLEN md:hist:binance:spot:BTCUSDT:1

# Состояние всех чанков
redis-cli HGETALL md:chunks:config
```
