## Боттлнеки

### 1. Redis pipeline — основной боттлнек на запись

**Что:** Каждые 50ms каждый коллектор делает `pipeline.execute()` с N командами `HSET`. При 8 запущенных коллекторах суммарно до 160 pipeline-вызовов в секунду.

**Проблема:** При большом числе символов (500+) и высоком tick rate пайплайн может накапливать отставание. Если `pipeline.execute()` занимает >50ms — следующий flush уже стоит в очереди.

**Признаки:**
- `proc latency p95 > 10ms` в снапшоте коллектора
- `t.writes/s` заметно ниже `messages/s`

**Решение:**
- Проверить `redis-cli --latency` на хосте Redis
- Перенести Redis на localhost, если сейчас сетевой
- Увеличить `TICKER_FLUSH_INTERVAL` до 100ms в `common.py`

---

### 2. WebSocket переподписка при реконнекте

**Что:** Bybit и Gate при реконнекте заново отправляют сотни subscribe-сообщений батчами.

**Проблема:** Пока идёт подписка (несколько секунд), данные не поступают. При нестабильной сети — регулярные дыры в истории и пропущенные арбитражные возможности.

**Признаки:**
- В снапшоте частые `reconnects > 0`
- Пропуски в `md:hist:bybit:*` списках

**Решение:**
- Мониторить `reconnects_total` в снапшоте
- При систематических дисконнектах — проверить стабильность сети
- Для Gate уменьшить `SUB_BATCH` если биржа отвечает таймаутом

---

### 3. Конвертация символов OKX/Gate — пропуски при нестандартных quote

**Что:** `_to_okx()` и `_to_gate()` возвращают `None` для символов не на USDT/USDC.

**Проблема:** Если в subscribe-файле есть `BTCBTC`, `ETHBNB` и т.п. — они **молча пропускаются**. Данные по ним не собираются.

**Признаки:**
- `symbols_tracked` меньше числа строк в subscribe-файле
- Нет ключа `md:okx:spot:XYZBTC` в Redis

**Решение:**
- Держать в subscribe-файлах только USDT/USDC пары (dictionaries обеспечивает это автоматически)

---

### 4. Memory growth при высоком tick rate

**Что:** `_history_buf[symbol]` накапливает записи между flush-циклами (каждые 1s).

**Проблема:** При зависании history flusher (Redis недоступен) буфер растёт бесконтрольно.

**Признаки:**
- RSS процесса растёт > 200MB
- `h.entries/s` = 0 при ненулевом `messages/s`

**Решение:**
- Контролировать RSS: `ps aux | grep python`
- Метрики RSS видны в redis_monitor через `memory.rss_bytes`

---

### 5. Binance — отсутствие RTT метрики

**Что:** Binance `bookTicker` не включает exchange timestamp.

**Проблема:** Нельзя измерить реальную задержку от биржи до скрипта. Видна только `proc latency` (буфер → Redis).

**Решение:** Для мониторинга сетевой задержки Binance — использовать внешний пинг до `stream.binance.com`. Для сравнения: Bybit, OKX, Gate предоставляют exchange ts.

---

### 6. Gate.io — задержка при подписке (SUB_BATCH × 50ms)

**Что:** Gate требует 50ms паузы между батчами subscribe.

**Проблема:** При 500 символах и `SUB_BATCH=100` — 5 батчей × 50ms = **250ms** до получения первых данных по последним символам после реконнекта.

**Это нормальное поведение.** При необходимости увеличить `SUB_BATCH` если биржа допускает.

---

### 7. signal_scanner — устаревшие данные

**Что:** Сканер читает из Redis данные, записанные коллекторами. Возраст данных (`data_age_ms`) показывает насколько свежи цены в момент принятия решения.

**Проблема:** Если коллектор отключён или Redis перегружен — сканер может принимать решения по устаревшим ценам.

**Признаки:**
- `data_age_ms.p95 > 500ms` в снапшоте signal_scanner

**Решение:**
- Проверить статус коллектора в снапшоте run.py
- Сканер не имеет встроенного TTL-отсева; при необходимости — добавить фильтр по `data_age_ms` в логике

## Нагрузочные тесты

Тесты находятся в `market_data/tests/load_test.py`. Запуск:

```bash
python3 market_data/tests/load_test.py
```

### Что тестируется

1. **Redis pipeline throughput** — скорость записи N тикеров через pipeline
2. **Redis read throughput** — скорость чтения HGETALL для N символов
3. **History RPUSH throughput** — скорость записи в List (история)
4. **SCAN throughput** — скорость сканирования `md:*` ключей
5. **Concurrent write** — параллельная запись из K корутин
6. **Chunk rotation** — запись + ротация чанков (реальный паттерн)

### Интерпретация результатов

| Тест                    | Норма                | Проблема           |
|-------------------------|---------------------:|--------------------|
| Pipeline write (500 sym)| < 5ms                | > 20ms             |
| Read HGETALL (500 sym)  | < 10ms               | > 50ms             |
| History RPUSH (1000 ent)| < 10ms               | > 30ms             |
| SCAN md:* (10k keys)    | < 50ms               | > 200ms            |
| Concurrent write (8×500)| < 15ms/batch         | > 50ms/batch       |

Если тест показывает числа в зоне "Проблема":
- `redis-cli --latency` — проверить базовую задержку
- `ping REDIS_HOST` — проверить сетевой путь
- `redis-cli INFO stats` — загрузка CPU/IO Redis-сервера

## Мониторинг в продакшне

### Ключевые метрики для алертинга

```
1. reconnects_total растёт > 5/мин     → нестабильный WS (коллектор)
2. proc latency p95 > 10ms             → Redis перегружен (коллектор)
3. msgs/s = 0 при статусе ACTIVE       → коллектор завис
4. symbols_active/symbols_tracked < 90%→ часть символов не получает тики
5. data_age_ms.p95 > 500ms             → устаревшие данные в сканере
6. redis_monitor: PING p95 > 10ms      → задержка сети до Redis
7. redis_monitor: rejected_connections → Redis отвергает соединения
```

### Проверка через redis-cli

```bash
# Сколько активных тикеров (должно ≈ суммарно символов по всем скриптам)
redis-cli --scan --pattern 'md:*:*:*' | grep -v hist | grep -v chunks | wc -l

# Последнее обновление конкретного тикера
redis-cli HGET md:binance:spot:BTCUSDT ts

# Размер истории для символа
redis-cli LLEN md:hist:binance:spot:BTCUSDT:1

# Состояние всех чанков
redis-cli HGETALL md:chunks:config

# Текущая нагрузка Redis
redis-cli INFO stats | grep -E "ops_per_sec|input_kbps|output_kbps"
```

### Проверка состояния через run.py

Снапшот каждые 10 секунд показывает uptime и статус каждого процесса. Признаки проблем:

```
↺ binance_spot     restarting in ~4s  restarts=3   ← частые рестарты
✗ okx_futures      stopped  exit=1  restarts=7     ← упал и не поднимается
```
